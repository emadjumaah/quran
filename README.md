# Quran Knowledge Graph (QKG)

A research-grade linguistic map of the Quran in a single SQLite database:
**every surah, ayah, word, morphological segment, and letter is a first-class,
queryable entity** with full grammatical metadata (root, lemma, part of speech,
verb form, aspect, mood, voice, case, person/gender/number, …) and full
structural positioning (juz, hizb, rub, ruku, page, manzil, sajda).

```
┌────────────────────────────────────────────────────────────┐
│ quran-kg.db                                                │
│   114 surahs → 6,236 ayahs → 77,429 words →                │
│   130,030 segments → 330,709 letters                       │
│   1,651 roots · 4,776 lemmas · 45 POS categories           │
│   + translations · + Gemini embeddings (optional layers)   │
└────────────────────────────────────────────────────────────┘
```

Audited end-to-end by a 20-agent adversarial review — see [AUDIT.md](AUDIT.md).
Architecture and entity model — see [ARCHITECTURE.md](ARCHITECTURE.md).

## Files

| File | What it is |
|---|---|
| `quran-kg.db` | **The deliverable.** Self-contained SQLite database (~52 MB). |
| `build_qkg.py` | Pipeline that builds `quran-kg.db` from `data/`. Deterministic — rerun any time. |
| `qkg.py` | Command-line explorer (`root`, `word`, `ayah`, `find`, `stats`). |
| `export_graph.py` | Exports node/edge CSVs (+ Neo4j Cypher) into `graph/` for Gephi/Cytoscape/networkx. |
| `queries.sql` | 18 ready-to-run example queries showing what the database can answer. |
| `quran-app.db` | App-shaped [monlite](https://github.com/qataruts/monlite) database (collections + FTS + root-relation edges + stats) generated from `quran-kg.db`. |
| `js/` | TypeScript workspace: the `quran-kg` npm package, **Quran Studio** (the React app), converters, and the Gemini embedding pipeline. See `js/README.md`. |
| `add_translation.py` | Enrichment: attach any Tanzil translation to `ayah_id` (default: Saheeh International). |
| `AUDIT.md` / `ARCHITECTURE.md` | The correctness audit record and the system architecture + entity model. |
| `graph/` | Node/edge CSVs (+ Neo4j Cypher) for network visualization, generated by `export_graph.py`. |
| `data/` | The four source files the build reads (see Provenance below). |
| `archive/` | Original project files (conversations, prompts, research PDFs, raw texts). Not used by the application. |
| `NEXT-STEPS.md` | Architecture feedback and the roadmap (npm publish, semantic layer, enrichment). |

## Quick start

```bash
python3 build_qkg.py            # rebuild the database (validates itself)
python3 add_translation.py      # attach the English translation layer
python3 qkg.py stats            # global statistics
python3 qkg.py root رحم         # every derivation + location of a root
python3 qkg.py word 2:255:5     # full morphology of one word
python3 qkg.py ayah 1:5         # word-by-word trace + translation
sqlite3 -box quran-kg.db < queries.sql   # run the example queries
python3 export_graph.py        # CSVs for network visualization

# the app layer (see js/README.md)
cd js && pnpm install
node scripts/convert-to-app-db.mjs       # quran-kg.db -> quran-app.db (monlite)
cd apps/studio && pnpm dev               # Quran Studio in the browser
```

## Schema

**Entities** (each row has a stable ID and a `location` like `2:255:5:2`):

- `surah` — names (Arabic/transliterated/English), Meccan/Medinan, traditional
  revelation order, bismillah flag, word/letter counts.
- `ayah` — Uthmani text + clean text, juz, hizb, rub (quarter), ruku, page
  (Madani mushaf), manzil, sajda type, word/letter counts.
- `word` — Uthmani form (exact orthography), clean form (searchable), the
  stem's root/lemma/POS denormalized for convenience, segment count.
- `segment` — the atom of the graph. Each word splits into prefix/stem/suffix
  segments; each carries: fine POS (45 tags), root, lemma, verb form (I–XII),
  aspect (perfect/imperfect/imperative), mood (indicative/subjunctive/jussive),
  voice, case, definiteness, person/gender/number, derivation (participles,
  verbal nouns), special family (إنّ / كان / كاد sisters), plus the raw
  source feature string for full fidelity.
- `letter` — every bare letter with word/ayah position (letter statistics).
- `root`, `lemma` — lookup tables with occurrence counts (lemma → root linked).
- `pos_tag`, `feature_glossary` — every tag explained in English and Arabic.
- `provenance` — where each layer of data came from.

**Views**: `v_word`, `v_segment` (fully decoded, human-readable), `v_root_freq`,
`v_lemma_freq`, `v_word_freq`, `v_letter_freq`, `v_surah_stats`, `v_root_map`.

Words sharing a root, lemma, pattern, or any grammatical property are connected
by construction — a join, not a search. See `queries.sql` for root maps,
co-occurrence networks, Meccan/Medinan contrasts, hapax legomena, verb-form
distributions, and more.

## Data provenance

| Layer | Source | Version / License |
|---|---|---|
| Morphology (roots, lemmas, POS, features) | [Quranic Arabic Corpus](https://corpus.quran.com) (Kais Dukes, Univ. of Leeds), Arabic-script edition via [mustafa0x/quran-morphology](https://github.com/mustafa0x/quran-morphology) | v0.4, GPL |
| Structural metadata (juz/hizb/ruku/page/sajda, surah names & types) | [Tanzil project](https://tanzil.net/docs/quran_metadata) | v1.0, CC BY 3.0 |
| Uthmani text | [Tanzil](https://tanzil.net/download) | v1.1, CC BY 3.0 |
| Clean text | project source `quran-clean.txt` (Tanzil simple-clean) | — |

The build validates itself on every run: canonical counts (114 surahs, 6,236
ayahs, 77,429 words, 130,030 segments), referential integrity, POS coverage,
and 100% alignment between Uthmani and clean word forms (including the
orthographic fusions «يا أيها», «بعدما», «يبنؤم», «ألو», «إل ياسين»).

## Regenerating from scratch

Only `data/` and the two Python scripts are needed. `build_qkg.py` re-creates
`quran-kg.db` deterministically in one command with no dependencies beyond
Python 3 stdlib.
